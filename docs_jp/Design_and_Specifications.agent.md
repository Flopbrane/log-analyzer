# Design and Specifications Agent Notes

> Target version: **v1.0.0-rc1**  
> This document is for future maintainers, assistants, and code-editing agents working on Logger_Project / Info Logger.

---

## 1. Project Identity

Info Logger is not a simple logging utility.

It is a structured logging, Event analysis, TraceQL search, FV recipe, summary, and investigation-report system.

The central rule is:

```text
LogDict = record
Event   = meaning
```

Do not collapse this distinction.

---

## 2. Current Release Baseline

The current baseline is:

```text
v1.0.0-rc1
```

This version assumes the following core pipeline works:

```text
read logs
  ↓
validate to LogDict
  ↓
summarize to Event
  ↓
Viewer display
  ↓
TraceQL/Event-aware search
  ↓
summary_engine
  ↓
CSV / JSON / investigation report export
```

---

## 3. Non-Negotiable Data Model Rules

### 3.1 `level`

`level` is the original severity recorded by the logger.

Examples:

```text
DEBUG
INFO
WARNING
ERROR
CRITICAL
REBOOT
```

### 3.2 `type`

`type` is analysis meaning.

Examples:

```text
TRACE_JUMP
REBOOT
REPEAT_ERROR
```

`type` may be `None` for normal logs.

### 3.3 `message`

`message` is what happened.

Examples:

```text
system_cpu_percent
reload_log prepared
trace_id changed
```

Do not put arbitrary message text into `type`.

Allowed message-to-type normalization must only return known `EventType` or `LogLevel` names/values. Unknown messages must return `None`.

---

## 4. Viewer Display Rule

The Type column must follow:

```python
if event.type is not None:
    display_type = event.type.name
else:
    display_type = event.level.name
```

If the code uses `normalize_message_type_name()`, it must only accept known values and must fall back to `event.level.name`.

---

## 5. Viewer State Responsibilities

The Viewer state should remain separated.

```text
raw_rows
    all validated source LogDict rows

filtered_rows
    LogDict rows corresponding to displayed Events

display_rows
    the only Event list used by Treeview, detail, CSV, JSON, and report output

_event_cache
    cached summarize(raw_rows) result
```

When raw logs are replaced:

```python
self.raw_rows = logs
self.filtered_rows = []
self.display_rows = []
self._event_cache = None
```

Any edit that bypasses cache invalidation is dangerous.

---

## 6. Event Cache Rule

Do not call `summarize(self.raw_rows)` repeatedly from multiple Viewer paths unless necessary.

Use:

```python
self._get_event_cache()
```

Expected behavior:

```text
first use:
    rebuild cache

same logs:
    reuse cache

new logs:
    clear cache and rebuild
```

---

## 7. Event-aware Search Rule

`type:TRACE_JUMP` must search Events, not raw logs.

Reason:

```text
TRACE_JUMP is generated by analysis.
It may not exist as raw text in LogDict.
```

The same applies to:

```text
type:REBOOT
type:REPEAT_ERROR
```

For original severity, use:

```text
level:ERROR
```

---

## 8. FV Recipe Rule

FV recipe execution must respect Event-aware queries.

Recommended flow:

```text
parse_fv_text()
  ↓
build_execution_plan()
  ↓
set Viewer search_var
  ↓
apply_filter()
  ↓
if query uses Event type:
      build FVResult from display_rows
   else:
      run normal execution_plan on raw_rows
```

Do not let `type:TRACE_JUMP` return zero just because it is not present in raw logs.

---

## 9. Summary Rule

`summary_engine` should remain GUI-independent.

Viewer should pass selected logs through a bridge/helper, such as:

```text
summary_bridge
```

Summary result may include:

```text
total_count
condition_text
level_counts
module_ranking
message_ranking
context_numeric_stats
insights
text
```

---

## 10. Export Rule

Export should preserve both records and analysis.

JSON bundle / investigation report should include:

```json
{
  "logs": [],
  "events": [],
  "summary": {},
  "report": {}
}
```

Investigation report metadata should include at least:

```json
{
  "kind": "investigation_report",
  "format_version": "0.1",
  "condition_text": "...",
  "timezone": "...",
  "log_count": 0,
  "event_count": 0,
  "source_files": []
}
```

---

## 11. Logger Internal Safety

Never call normal logger methods inside logger internals.

Forbidden places include:

```text
_log
_build_log_record
_safe
_emit
```

Recursive logging can cause infinite loops.

Use `print()` or special internal handling for logger-internal warnings.

---

## 12. Time Rule

UTC is the internal truth.

```text
recording: UTC
storage: UTC
analysis: UTC
display: selected local timezone
```

Viewer timezone changes should not mutate stored logs.

---

## 13. TraceQL Boundary Rule

Logger/UI code should not directly depend on `query_engine`.

Use:

```text
logs.traceql_bridge
```

Boundary:

```text
logs / viewer
  ↓
logs.traceql_bridge
  ↓
query_engine
```

---

## 14. Query Error Rule

Invalid TraceQL must not crash the Viewer.

It should become a diagnosable query error report with:

```text
query text
error message
caret position
expected syntax
dictionary-based suggestions
```

AI is not required for query suggestions.

---

## 15. File Adapter Rule

External log formats should be normalized into `LogDict` before entering main analysis.

Do not let external formats leak into Viewer-specific code.

Preferred flow:

```text
external file
  ↓
file_adapter
  ↓
list[dict]
  ↓
log_validator
  ↓
list[LogDict]
```

---

## 16. Large Log Rule

Current Viewer is correctness-first.

For large logs, prefer future work in:

```text
SQLite adapter
batch search
paging
indexed search
lazy Event generation
```

Do not prematurely break Event correctness for speed.

## 19. Release Checklist

Before tagging a stable release:

```text
□ open single log
□ open multiple logs
□ reload another log and confirm cache clears
□ search type:TRACE_JUMP
□ search type:REBOOT
□ search level:ERROR
□ open detail window
□ run FV recipe
□ show SummaryWindow
□ export JSON bundle
□ export investigation report
□ verify source_files and report metadata
```

---

## 20. Maintenance Warning

Avoid "quick fixes" that make the Type column display arbitrary message text.

That was a previous failure mode.

Keep this model:

```text
level   = original severity
type    = analysis meaning
message = what happened
```

This is the heart of the current Viewer.
